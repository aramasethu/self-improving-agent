# Self-Improving Data Migration Agent

A LangGraph agent that takes messy CSV employee data and maps it to a standardized 6-field HR schema. It then learns from its own failures using LangSmith traces, generating rules that get injected back into future prompts.

## Setup

```bash
cp .env.example .env   # add ANTHROPIC_API_KEY and LANGSMITH_API_KEY
uv sync
```

## Folder Structure

- `data/test/` - 110 synthetic test CSVs with varied column names, date formats, salary formats, and distractor columns. Generated by scripts in `scripts/`.
- `data/ground_truth.json` - Golden dataset with the correct column mapping and expected output for each CSV. Also generated by scripts.
- `results/agent_outputs/` - JSON output from the production agent (one file per CSV).
- `results/agent_status_quo/` - JSON output from the baseline agent.
- `results/agent_improved/` - JSON output from the improved agent.
- `results/evaluations/` - Timestamped evaluation result files.
- `rules/` - Learned improvement rules (`improvement_rules.json`), generated by analyzing failure patterns.
- `scripts/` - Utility scripts for generating synthetic test data (`generate_datasets.py`, `add_hard_csvs.py`), running multi-iteration experiments (`run_iterations.py`), and comparing evaluation runs (`compare_runs.py`).

## Running the Demo

All examples below use `data/test/test_100.csv`, a CSV with mixed salary formats that reliably causes failures in the baseline agent.

### 1. Run the baseline agent (no tracing, no rules)

```bash
uv run agent_status_quo.py data/test/test_100.csv
cat results/agent_status_quo/test_100.json | python -m json.tool
```

### 2. Evaluate baseline results

```bash
uv run agent_status_quo.py
uv run python -c "from evaluate import run_evaluation; run_evaluation(agent_output_dir='results/agent_status_quo')"
```

### 3. Run with LangSmith tracing

```bash
uv run agent_status_quo_traced.py data/test/test_100.csv
```

Traces appear in the LangSmith UI under the project `status-quo-baseline`.

### 4. Analyze failures and generate rules

```bash
# Reset rules
uv run python -c "
import json, os
os.makedirs('rules', exist_ok=True)
with open('rules/improvement_rules.json', 'w') as f:
    json.dump({'rules': [], 'metadata': {'last_analysis': '', 'total_runs_analyzed': 0, 'version': 0}}, f, indent=2)
"

# Analyze
uv run python -c "from analyze_traces import run_analysis; run_analysis(min_failures=1, agent_output_dir='results/agent_status_quo')"

# Check the generated rules
cat rules/improvement_rules.json | python -m json.tool
```

### 5. Run improved agent with rules

```bash
uv run agent_improved.py
uv run python -c "from evaluate import run_evaluation; run_evaluation(agent_output_dir='results/agent_improved')"
```

## Self-Improvement Loop

```
run agent -> evaluate -> analyze failures -> generate rules -> validate rules -> inject into prompts
```

Rules are stored in `rules/improvement_rules.json` and injected into the code generation prompt on subsequent runs. A keep/discard gate compares composite scores before and after, reverting rules that make things worse.

## Evaluation Metrics

- **Schema accuracy**: does the column mapping match the golden dataset?
- **Execution success**: does the generated code run without errors?
- **Field correctness**: are output values valid (dates as YYYY-MM-DD, salaries as floats, emails with @)?
- **Composite score**: `0.3 * schema + 0.3 * execution + 0.4 * fields`

## Data Generation

The test CSVs and golden dataset are synthetic, generated by `scripts/generate_datasets.py` (100 base CSVs) and `scripts/add_hard_csvs.py` (10 additional harder cases with mixed salary formats and monthly salaries). One sample CSV (`data/test/test_100.csv`) is committed to the repo for quick testing. To regenerate all data, run those scripts.
